# 단어 시험 채점 웹앱 설계

## 배경

기존에는 Claude Code로 Word.xlsx(단어장/보카테스트/채점결과 탭)를 직접 열어서 채점을 진행했다. 여러 반 선생님이 각자 쓰려면 웹 도구가 필요하다. 반 명단 기반 문서 자동화(출석부/교재배부명단/관리일지) 기능은 별도 설계 문서에서 다룬다.

## 핵심 제약

1. 개인정보(학번, 연락처 등)는 서버 어디에도 저장하지 않는다. 요청 처리 중 메모리에만 존재하고 응답 후 폐기한다.
2. 반마다 단어장(정답 리스트)이 다르며, 이건 개인정보가 아니므로 저장해두고 채점 시 불러 쓴다.
3. AI가 없는 정보나 사실을 추가하지 않는다 — 단어장에 나열된 뜻 외의 뜻을 정답으로 인정하지 않는다.
4. 모든 결과물은 정확해야 하며, 애매한 판단은 검토 가능해야 한다.
5. 결과물은 복사/붙여넣기로 쓸 것이므로 붙여넣기 편한 형태로 제공한다.

## 아키텍처

- Next.js 앱을 Vercel에 배포. 프론트엔드와 서버리스 API 라우트를 한 프로젝트에 둔다.
- 단어장은 Vercel Blob에 반+데이 단위 파일로 저장한다.
- 학생 답안·이름 등은 요청 처리 중에만 서버 메모리에 존재하고, 응답 후 저장하지 않는다.
- OpenAI API 키는 서버리스 함수 안에서만 사용하고 브라우저에 노출하지 않는다.
- 사이트 진입 시 공유 비밀번호 1개로 접근을 제한해 익명 사용/비용 낭비를 막는다.

## 화면 구성

1. **비밀번호 입력 화면**: 공유 비밀번호 통과 후 진입.
2. **반+데이 선택 화면**: 드롭다운으로 반과 데이(일차)를 선택.
3. **단어장 관리 화면**: 선택한 반+데이의 단어장(영단어 + 뜻1~4)을 업로드/수정. 기존 등록 내용이 있으면 보여주고 덮어쓰기 가능.
4. **답안 입력 화면**: 구글시트 링크 붙여넣기 또는 엑셀/CSV 업로드 중 선택. 이름 + 답 30개가 순서대로 있다고 가정(단어장 순서와 1:1 대응).
5. **결과 화면**:
   - 상단: AI가 판단에 확신이 없었던 애매한 항목만 이유와 함께 표시. 교사가 그 자리에서 정답/오답을 뒤집을 수 있다.
   - 하단: 최종 결과 표 `이름 | 틀린 단어` (학번은 포함하지 않음). 같은 단어를 10명 이상이 틀리면 표 맨 위에 경고 행으로 `단어 | 오답 인원 | 공통 오답`을 추가한다.
   - 표는 드래그 선택 시 탭 구분자로 복사되어 엑셀에 바로 붙여넣기 가능하게 만든다.

## 데이터 흐름 (채점 1회 기준)

1. 사용자가 반+데이 선택 → 서버가 Vercel Blob에서 해당 단어장을 불러온다.
2. 사용자가 답안(구글시트 링크 또는 파일)을 제출 → 서버가 즉시 파싱한다 (이름 + 답 30개, 순서 매칭).
3. 서버가 학생별로 "단어 + 정답뜻(전부) + 학생답" 묶음을 만들어 OpenAI에 배치 요청한다 (학생 1명당 API 요청 1번).
4. OpenAI 응답(정답/오답 + 애매한 항목은 판단 이유)을 서버가 취합한다.
5. 반 전체 학생 기준으로 단어별 오답 인원을 집계해, 10명 이상이면 경고 목록에 추가한다.
6. 결과를 브라우저에 반환하고 표시한다. 서버는 이 요청 처리가 끝나면 답안/이름 데이터를 메모리에서 버린다 (저장 없음).

## 채점 판정 기준

기존 Word.xlsx 채점에서 확정된 기준을 그대로 따른다.

- 표현만 다르고 품사·의미가 같으면 정답 (예: "잠재적인" vs "잠재의").
- 품사가 명백히 다르면 오답 (예: "밝아지다"(동사) vs "밝다"(상태)).
- 복합명사/구의 일부 의미 요소가 빠지면 오답 (예: "느리게 읽는 사람" vs "느리다").
- 단어장에 명시적으로 나열되지 않은 뜻은 정답으로 인정하지 않는다.
- 미응답(빈칸)은 자동 오답 처리한다.

## 오류 처리

- 선택한 반+데이의 단어장이 없으면 "먼저 단어장을 등록해주세요" 안내.
- 답안 개수가 단어장 단어 수와 맞지 않으면 경고 표시 (순서 어긋남 가능성).
- OpenAI 응답 형식이 깨지면 해당 학생만 재시도하고, 그래도 실패하면 "이 학생은 수동 확인 필요"로 표시한다 (전체 채점은 막지 않는다).

## 테스트 방향

- 채점 프롬프트/판정 로직은 이번 대화에서 이미 확정된 실제 사례(박성재/이주호/김대연, DAY 04)를 픽스처로 활용해 검증한다.
- OpenAI 호출부는 목(mock) 응답으로 정상/에러 케이스를 각각 테스트한다.
- 단어장 개수 불일치, 미응답 처리, 10명 이상 공통 오답 집계 로직은 별도 단위 테스트로 검증한다.

## 범위 밖 (이번 스펙에서 다루지 않음)

- 이미지 답안지 채점 (추후 별도 논의).
- 반 명단 기반 문서 자동화 (출석부/교재배부명단/관리일지) — 별도 설계 문서에서 다룸.
- 학번 표시 (이번 라운드에서는 이름만 출력하기로 결정).

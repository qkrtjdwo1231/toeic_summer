# 단어장 입력 화면 개선 설계

## 배경

배포된 `/grading` 기능에서 사용자가 3가지 문제를 제보했다.

1. 단어장(정답지)을 이미지나 엑셀 파일로 업로드하고 싶다.
2. 학생 답의 띄어쓰기 유무로 오답 처리되는 버그. → 이미 원인 분석 및 수정 완료(2026-07-08, `lib/grading/grade-student.ts`의 채점 기준 문구에 예외 규칙 추가, 커밋 `3b496bb`). 이 문서의 범위 밖.
3. 다중 뜻(한 단어에 뜻이 여러 개) 단어의 채점 미숙. → 채점 시점 문제는 위와 같은 프롬프트 수정으로 이미 해결됨. 다만 논의 중 "단어장 입력 화면 자체를 표 형식으로 나누면 다중 뜻 인식(이미지/엑셀 자동 인식) 정확도가 더 올라갈 것 같다"는 제안이 나왔고, 이를 반영하기로 했다.

이 문서는 1번(파일 업로드)과, 그 과정에서 나온 단어장 입력 화면의 표 형식 전환을 다룬다.

## 핵심 제약

1. 단어장은 개인정보가 아니므로 (기존 정책과 동일) 저장/전송에 특별한 제약이 없다.
2. 새로 설치할 npm 패키지가 없다 — 이미지 인식은 이미 설치된 `openai` 패키지(gpt-4o-mini의 이미지 인식 기능)를, 엑셀/CSV 읽기는 이미 설치된 `xlsx` 패키지를 그대로 쓴다.
3. AI가 이미지를 잘못 읽을 수 있으므로, 파일 업로드 결과는 화면의 입력 표에 채워지기만 하고 **자동 저장되지 않는다**. 선생님이 눈으로 확인/수정한 뒤 기존과 동일하게 "저장하고 다음" 버튼을 눌러야 실제로 저장된다.
4. 한 단어당 인정되는 뜻은 최대 5개로 늘린다 (기존 4개 → 5개).

## 화면 구성 변경

### 기존

- 단어장 입력 화면에 큰 `<textarea>` 하나. `영단어[Tab]뜻1[Tab]뜻2...` 형식으로 한 줄에 한 단어씩 직접 입력.

### 변경 후

- 실제 `<table>` 입력창으로 전환. 열 구성: `영단어 | 뜻1 | 뜻2 | 뜻3 | 뜻4 | 뜻5 | 삭제`.
- 각 셀은 독립된 입력 상자(input) — 탭 문자를 몰라도 되고, 뜻이 뒤섞이는 실수가 원천 차단된다.
- 표 아래 "단어 추가" 버튼으로 빈 행을 늘릴 수 있고, 각 행의 "삭제" 버튼으로 그 행을 지울 수 있다.
- 기존에 저장된 단어장을 불러오면(반+데이 선택 후) 자동으로 표에 행이 채워진다. 새로 시작하면 빈 행 1개로 시작한다.
- 표 위(또는 옆)에 "파일에서 불러오기" 버튼 하나를 추가한다. 이미지(jpg/png)와 엑셀/CSV를 모두 받는 파일 선택창이며, 서버가 파일 종류를 자동으로 구분해서 처리한다 (버튼을 이미지용/엑셀용으로 나누지 않는다).

## 데이터 흐름

### 표 상태 관리 (화면 내부)

- 화면(`app/grading/page.tsx`)의 상태를 "긴 문자열 하나"에서 "행 배열"(`{ word: string; meanings: string[] }[]` 또는 동등한 6칸 문자열 배열)로 바꾼다.
- 저장 시 이 행 배열을 그대로 기존 `PUT /api/grading/wordlist` API에 보낸다 (API가 이미 `rows: string[][]`를 받고 있어 API 자체는 변경 없음).

### 파일 업로드 → 표 채우기

1. 선생님이 "파일에서 불러오기"로 이미지 또는 엑셀/CSV 파일을 선택한다.
2. 화면이 새 API `POST /api/grading/wordlist/extract`에 파일을 보낸다.
3. 서버는 파일 종류(MIME 타입)를 보고 분기한다.
   - 엑셀/CSV → `lib/grading/wordlist-file.ts`의 엑셀 파싱 함수로 표 형태(행×열)를 읽는다. A열=단어, B~F열=뜻1~5로 가정한다 (화면 표의 열 순서와 동일).
   - 이미지(jpg/png) → `lib/grading/wordlist-ocr.ts`가 OpenAI(gpt-4o-mini)에 이미지를 보내 "사진 속 단어장을 읽어서 단어별 뜻(최대 5개)을 JSON 배열로 알려줘"라고 요청하고, 응답을 같은 행 형태로 변환한다. 사진마다 단어당 뜻 개수가 다를 수 있으므로 모자란 칸은 빈 칸으로 채운다.
4. 서버는 어느 경로든 결과를 `{ rows: [[단어, 뜻1, 뜻2, 뜻3, 뜻4, 뜻5], ...] }` 형태로 응답한다.
5. 화면은 이 `rows`를 받아 표의 행들을 **통째로 교체**한다 (기존 입력 내용은 버려짐 — 사용자 확인 완료된 결정).
6. 선생님이 표를 확인/수정 후 기존과 동일하게 저장 버튼을 누른다.

## 코드 변경 범위

**수정**
- `lib/grading/wordlist.ts`: `parseWordListRows`가 뜻을 최대 4개(`row.slice(1, 5)`)까지만 받던 것을 5개(`row.slice(1, 6)`)까지 받도록 변경.
- `app/grading/page.tsx`: 단어장 입력 UI를 텍스트 칸 → 표로 전환, 파일 업로드 버튼 추가.

**신규**
- `lib/grading/wordlist-file.ts`: 엑셀/CSV 버퍼를 읽어 `[단어, 뜻1~5]` 행 배열로 변환 (기존 `answers-file.ts`와 유사한 패턴).
- `lib/grading/wordlist-ocr.ts`: 이미지 버퍼 + OpenAI 클라이언트를 받아 OCR 결과를 같은 행 배열로 변환.
- `app/api/grading/wordlist/extract/route.ts`: 파일을 받아 위 두 함수 중 하나로 분기 처리 후 `rows`를 반환하는 API.

## 오류 처리

- 이미지 인식 실패(AI 응답이 JSON이 아니거나 형식이 안 맞음): `{ error: "이미지를 인식하지 못했습니다. 다시 시도하거나 직접 입력해주세요." }` 응답. 화면은 표를 그대로 두고(교체하지 않음) 에러 메시지만 보여준다.
- 엑셀 형식이 이상한 경우(단어 칸이 모두 비어 있는 등 유효한 행이 하나도 없음): `{ error: "엑셀 형식을 확인해주세요." }` 응답. 마찬가지로 표는 그대로 둔다.
- 지원하지 않는 파일 형식(이미지/엑셀/CSV가 아님): `{ error: "이미지 또는 엑셀/CSV 파일만 업로드할 수 있습니다." }` 400 응답.
- 어느 경우든 실패해도 화면이 깨지거나 기존에 입력하던 표 내용이 사라지지 않는다.

## 테스트 방향

- `lib/grading/wordlist.ts`: `parseWordListRows`가 5번째 뜻(인덱스 5)까지 정확히 잘라내는지, 6번째 이후는 버리는지 테스트.
- `lib/grading/wordlist-file.ts`: 합성 엑셀 버퍼로 `[단어, 뜻1~5]` 행 추출 테스트, 단어 칸이 빈 행은 걸러내는지 테스트.
- `lib/grading/wordlist-ocr.ts`: OpenAI 클라이언트를 mock해서, 정상 JSON 응답 파싱 / 뜻 개수가 들쭉날쭉한 경우 빈 칸 채움 / JSON이 아닌 응답일 때 에러 처리를 테스트 (기존 `grade-student.test.ts`의 mock 클라이언트 패턴 재사용).
- API 라우트: 파일 없음/미지원 형식일 때 400 에러 테스트.
- 화면(표 UI, 행 추가/삭제, 파일 업로드 결과 반영)은 `npm run build` + 수동 확인으로 검증 (기존 화면 컴포넌트들과 동일한 테스트 정책).

## 범위 밖 (이번 설계에서 다루지 않음)

- 학생 답안 업로드(엑셀/CSV)는 이미 지원 중이며 이번 설계와 무관 — 이번 설계는 **단어장(정답지) 입력** 단계만 다룬다.
- 이미지 업로드 시 여러 장을 한 번에 합쳐서 인식하는 기능(한 번에 이미지 1장만 지원).
- 표의 행 순서를 드래그로 재배열하는 기능.
